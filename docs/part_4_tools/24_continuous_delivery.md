# Chapter 24 — Continuous Delivery

> *Parallel to:* SWE-book Chapter 24 *"Continuous Delivery"* (Winters, Manshreck, Wright 2020).

**Thesis.** CD is the practice of releasing every CI-green commit. The DSBench parallel is the **submission archive build** (`framework/build_submission.py`) plus the **`git push origin main`** that publishes the cohort state to <https://github.com/dlmastery/autoresearch_dsbench>. The submission archive is the project's release artefact; the GitHub push is the publication.

## 24.1 What gets delivered

Each cohort run produces one release. The release has three parts:

| Part | Location | Content |
|---|---|---|
| **Cohort state** | `registry/final_rollup.json` + `registry/forensic_summary.json` | 112 rows each: per-task test score, DSBench delta, BEAT verdict, forensic verdict. |
| **Per-task submission archive** | `submissions/dsbench_submission/<kind>/<slug>/` × 112 | 14 files per task: `README.md`, `config.json`, `final_report.json`, `runner_up.json`, `audit_report.md`, `forensic_audit.{md,json}`, `research_journal.md`, `experiment_summary.md`, `experiment_log.jsonl`, `reasoning_annotations.json`, `code/CLAUDE.md`, `inference/predict.py`, `reproduction/reproduce_log.txt`. |
| **Cross-task dashboard** | `dashboard/index.html` + `dashboard/md_viewer.html` + `dashboard/task_detail.html` | Sortable per-task scoreboard; per-task drilldown via two-tab navigation; inline Markdown rendering. |

The three parts are coherent: the dashboard reads the rollup; the rollup reads the submission archive; the submission archive contains everything a reviewer needs to spot-check a single task.

## 24.2 The submission-archive contract

The 14-file per-task archive is the smallest unit a reviewer would receive. Per `framework/build_submission.py`:

| File | Source | Purpose |
|---|---|---|
| `README.md` | Generated | Task summary + champion + DSBench delta + reproduction one-liner. |
| `config.json` | Mirror of `best_config.json` | Exact champion config (backbone + params + composite). |
| `final_report.json` | Mirror of `<task>/autoresearch_results/final_report.json` | One-shot test-set score, DSBench delta, beats verdict. |
| `runner_up.json` | Computed from `experiment_log.jsonl` | 2nd-best experiment as a variance baseline. |
| `audit_report.md` | Generated by build_submission | 14-section explainability audit. |
| `forensic_audit.md` + `.json` | Mirror | 10-agent committee report. |
| `research_journal.md` | Mirror | Per-experiment 6-field narrative. |
| `experiment_summary.md` | Mirror | Tabular per-experiment summary. |
| `experiment_log.jsonl` | Mirror | Append-only raw log. |
| `reasoning_annotations.json` | Mirror | Machine-readable per-experiment reasoning. |
| `code/CLAUDE.md` | Mirror | The task-specific protocol file. |
| `code/task_config.json` | Mirror | Frozen task identity. |
| `code/paper.md` + `paper_abstract.md` | Mirror | Task references. |
| `code/seed_reasoning.json` | Mirror | First-experiment plan. |
| `inference/predict.py` | Generated | Standalone inference script. Takes a config + a numpy array; returns predictions. |
| `reproduction/reproduce_log.txt` | Generated | Runnable PowerShell repro command + expected composite within ±0.005. |

The contract is *stable* — every cohort run produces the same 14-file shape per task. A reviewer who learnt the shape on one task knows it on all 112.

## 24.3 The build_submission command

```powershell
& "C:/Users/evija/anaconda3/python.exe" framework/build_submission.py [--task <slug>] [--all]
```

Behaviour:

- Without `--task`, builds every task that has a `final_report.json` and whose champion changed since the last build.
- With `--task <slug>`, builds one task.
- With `--all`, builds every task unconditionally (used after a framework change that affects the audit report).

The script is *idempotent*: running it twice produces the same per-file output (modulo timestamps in the README). The 14-section explainability report is regenerated from the experiment log + the champion config + the per-fold breakdown; the script does not re-train.

Wall-clock: ~30 seconds per task for the explainability report (which computes SHAP-style feature importance + calibration + uncertainty diagrams). Full-cohort `--all` run: ~30 minutes.

## 24.4 The dashboard build

`framework/_refresh_dashboards.py` produces:

- `dashboard/index.html` — the cross-task scoreboard. Two tabs (modeling / analysis). Sortable columns: `composite`, `val_score`, `train_score`, `test_score`, `dsbench_baseline`, `delta_vs_dsbench`, `beats_dsbench`, `forensic_verdict`. Links per row open the per-task detail.
- `dashboard/md_viewer.html` — a Markdown viewer. All `*.md` links in the dashboard route through this viewer so `forensic_audit.md` and `audit_report.md` render inline ([ADR 0011](../appendix_b_adrs/0011_md_viewer_inline_render.md)).
- `dashboard/task_detail.html` — the per-task detail page. Tabs for *Reasoning*, *Forensic Audit*, *Explainability Audit*, *Trade Logs*, *Backbone Tabs*. Implements the two-tab navigation pattern (Lessons-Learned row 23, [ADR 0012](../appendix_b_adrs/0012_two_tab_navigation.md)).
- `autoresearch_results/dashboard.html` per task — the legacy per-task dashboard. Preserved for backward compatibility.

The dashboard is *static HTML* — no server-side rendering. A reviewer can `python -m http.server 8765 --directory C:/Users/evija/dsbench` and navigate the local dashboard, or open the file directly. The static-only design means the dashboard works offline.

## 24.5 The GitHub push

The final delivery step: `git push origin main` to <https://github.com/dlmastery/autoresearch_dsbench>. Steps:

1. `framework/_final_audit.py` exits 0.
2. `git add -A docs/ submissions/ registry/ modeling/ analysis/ skills/ framework/ dashboard/ README.md STATUS.md`
3. `git commit -m "<message>"` with a body explaining the cohort change.
4. `git push origin main`.

The push is single-stream (no other branches). The schannel → openssl SSL backend swap ([postmortem 0005](../appendix_a_postmortems/0005_git_push_ssl_cert_failure.md)) is a one-time global config; once done, push works.

After the push, the GitHub repo's `README.md` + `STATUS.md` + the `dashboard/index.html` are the public face of the project.

## 24.6 What "release" means here

The SWE-book chapter 24 distinguishes pre-release (CI green), release (artefact published), and deployment (artefact running in production). Our analogues:

| SWE-book stage | DSBench analogue |
|---|---|
| Pre-release | `_final_audit.py` exits 0. |
| Release | `framework/build_submission.py --all` ran; 112 per-task archives exist in `submissions/`. |
| Publication | `git push origin main` succeeded. |
| Deployment | N/A — there is no production system. The reviewer's local copy *is* the deployment. |

The project does not ship a service. The cohort state lives on GitHub; a reviewer clones, reads, and audits.

## 24.7 The "rollback" contract

What happens if a release is bad? Three rollback paths:

1. **Per-task rollback.** If one task's submission is suspect, the human can `git checkout HEAD~1 -- submissions/dsbench_submission/<kind>/<slug>/` and rebuild from the previous champion config. The audit gate re-runs over the rebuilt state.
2. **Cohort-wide rollback.** If a framework change broke the cohort, `git checkout v<previous-tag> -- framework/` and re-run the cohort. This is the bisection-supported large-scale rollback.
3. **GitHub rollback.** If a push was wrong, `git revert <bad-commit>` and force-push is forbidden by convention. Always go forward with a corrective commit.

The five postmortems in [Appendix A](../appendix_a_postmortems/) each describe an incident; none required a full rollback. The closest was [postmortem 0001](../appendix_a_postmortems/0001_regression_delta_sign_bug.md): commit `b0b417e` had the wrong delta sign for 16 regression tasks; the corrective commit `1be5130` superseded it. Both commits remain in git history.

## 24.8 The "release notes" analogue

The project does not have formal release notes. The closest analogue:

- The **commit message body** on the cohort push names the scoreboard change ("BEAT-DSBENCH lifted from 70 → 82, FORENSIC PASS held at 112/112").
- The **Lessons-Learned table** in `framework/CLAUDE_template.md` lists the protocol-level changes since the previous cohort.
- The **STATUS.md** at the repo root carries the cohort scoreboard snapshot.

A reviewer reading these three together gets the full release-note experience.

## 24.9 The continuous-delivery cadence

The project's CD cadence has been roughly:

- Initial cohort: 2026-05-15 (commit `b0b417e`, 66/74 modeling BEAT — pre-sign-fix).
- Sign-fix + Lessons rewrite: 2026-05-15 (commit `1be5130`, 82/112 BEAT, 112/112 PASS).
- This documentation release: 2026-05-16 (this commit).

Cadence is event-driven, not time-driven. The project does not release on a calendar; it releases when the cohort state changes materially.
