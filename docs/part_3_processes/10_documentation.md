# Chapter 10 — Documentation

> *Parallel to:* SWE-book Chapter 10 *"Documentation"* (Winters, Manshreck, Wright 2020).

**Thesis.** Documentation is code that nobody runs but everybody reads. It is the longest-lived artefact the project produces and the one most likely to lie. The DSBench project treats documentation as code: it is regenerated by scripts, audited by the four-layer gate, and forbidden from making claims that have not been measured. The two anchor rules are the **Six-Field Annotation Rule** and the **No-Future-Sections Rule**.

## 10.1 The four documentation surfaces

Documentation in this project lives on four surfaces, each with a different reader, lifetime, and update cadence.

| Surface | Reader | Lifetime | Update cadence | Format |
|---|---|---|---|---|
| Per-task `CLAUDE.md` | Claude Code at session start | Project-long (regenerated on framework change) | Per framework change | Markdown, templated from `CLAUDE_template.md`, mechanically validated against `SECTION_MAPPING.md` |
| Reasoning annotations (`reasoning_annotations.json`) | Reviewer; next-session Claude | Per task, append-only | Per experiment | JSON, six fields per entry, `_manual: true` flag |
| Research journal (`research_journal.md`) + experiment summary | Human review; eventual reader | Per task, append-only | Per experiment | Markdown, generated from the JSON annotations |
| This docs/ (the SWE-book-style book) | Anyone onboarding to the project | Project-long | Per major release | Markdown, 25 chapters + 4 appendices |

Each surface has its own audit hook (validator, forensic-audit Agent E for annotations, dashboard refresh for the journal, manual review for this book). No surface is updated by hand without an audit hook.

## 10.2 The Six-Field Annotation Rule

The core unit of documentation in this project is the **per-experiment annotation**. Every experiment in every task carries an entry in `autoresearch_results/reasoning_annotations.json` keyed by experiment number:

```json
{
  "7": {
    "diagnosis": "...",
    "citations": "...",
    "hypothesis": "...",
    "prediction": "...",
    "verdict": "...",
    "learning": "...",
    "_manual": true
  }
}
```

Six fields, in two phases. The first four are **pre-write** (authored *before* the experiment runs):

- **diagnosis** — what is the current state, by reading the previous experiment + the per-fold breakdown. Cites the specific numerical evidence (e.g. "fold 3 lagging by 1.2 % AUC").
- **citations** — at least one paper in the canonical format (Author, YEAR, VENUE, 'Title', arXiv ID, one-sentence relevance).
- **hypothesis** — the mechanism by which the proposed change should help, in 1–3 sentences. Names the knob being changed.
- **prediction** — a *numerical* prediction of the composite delta. Vague predictions ("should be better") are rejected.

The last two are **post-write** (authored *after* the experiment runs):

- **verdict** — KEEP / DISCARD with the composite delta vs the prior champion.
- **learning** — what the experiment revealed about the loss landscape, even if it failed. Negative results are informative.

The rule has teeth: the `dashboard-reasoning-annotations` skill carries the canonical format text; the dashboard refuses to render an annotation with fewer than six fields; the forensic audit's Agent E spot-checks for completeness on the champion lineage.

### Worked example

From `titanic` exp 7 (xgboost depth-10 push):

```json
{
  "7": {
    "diagnosis": "Champion (iter-2 depth=8) achieved val=0.9520 with train/val gap 0.04. Per-fold breakdown shows fold-3 lagging by 1.2% AUC...",
    "citations": "Chen & Guestrin 2016 KDD 'XGBoost: A Scalable Tree Boosting System' (arXiv:1603.02754) — depth-vs-iter tradeoff section 4 argues...",
    "hypothesis": "Increase max_depth 8→10 with proportional lr drop 0.05→0.04 to keep per-tree contribution constant. Mechanism: deeper trees model 3rd-order interactions...",
    "prediction": "Composite should move from 0.9543 to approximately 0.9550-0.9580. If gap > 0.05 the iter is exhausted as a direction.",
    "verdict": "KEEP — composite 0.9555, +0.0012 vs champion. Fold-3 recovered as predicted; fold-1 marginally regressed.",
    "learning": "Depth=10 is the new ceiling for this task. Going to depth=12 next would require lr ≤ 0.03 to keep per-tree contribution bounded.",
    "_manual": true
  }
}
```

The `_manual: true` flag is meaningful. It says: a human (or a human-reviewed Claude session) wrote this. Champion lineage must be `_manual: true` end-to-end.

## 10.3 The No-Future-Sections Rule

> Documentation does not describe state that has not been measured. Numbers come from running code, not from estimation.

The rule plays out concretely in two places:

### 10.3.1 The Champion-summary placeholders

The top of every `CLAUDE.md` carries a "Champion summary" block:

```markdown
- **Beat DSBench:** <count>/<total> (<percent>%)
- **Forensic PASS:** <count>/<total>
- **Champion of champions:** <task_slug> · <backbone> · composite <value>
```

The placeholders are *literal*. They are not auto-substituted by the template renderer. A future pre-commit hook (`framework/_substitute_champion_summary.py`) will read `registry/final_rollup.json` + `registry/forensic_summary.json` and substitute the literals into actual counts *at the moment of scaffold generation*. Until that hook ships, the placeholders remain literal so the file never lies.

The rule says: if the substitution hook is not yet running, the placeholders stay. We do not write the numbers manually, because manual numbers drift within hours of being written.

### 10.3.2 The "future work" section

The template has no "roadmap" or "future work" section. The reason: the project does not promise things it hasn't built. If a thing is planned, it lives as a TODO in an ADR's Consequences section, not as a documentation claim. The reader of any per-task `CLAUDE.md` learns what *is*, never what *might be*.

This is the SWE-book chapter 10 rule "documentation must not drift" pushed to its strongest form: *forbid promises that drift can corrupt*.

## 10.4 Documentation as code

The four surfaces of documentation are all under version control. Three of the four are regenerated by scripts:

| Surface | Regenerated by |
|---|---|
| Per-task `CLAUDE.md` | `framework/generate_scaffolds.py` (or `framework/_regenerate_claude_only.py` for minor refreshes) |
| Reasoning annotations | `framework/runner.py` writes the file structure; Claude / human writes the content |
| Research journal | `framework/runner.py:_append_research_journal` |
| This docs/ | Human, but the structure is committed to and audited via link-check |

The dashboards are also documentation in the SWE-book sense — they're a presentation layer over the rollup files. `framework/_refresh_dashboards.py` is the regeneration script. Hand-editing a dashboard's HTML is forbidden; the next refresh would overwrite the edit.

## 10.5 The Citation-Rigor Rule

Documentation is hollow without references. The Citation-Rigor Rule (introduced in [Ch. 3](../part_2_culture/03_knowledge_sharing.md), enforced here):

> Every citation in the canonical format: `Author1, Author2, ... YEAR VENUE 'Paper Title' (arXiv:XXXX.XXXXX) — one-sentence relevance note`.

The rule is enforced in three places:

1. **The annotations.** Every experiment cites ≥ 1 paper in the canonical format.
2. **The chapter files** (this docs/). Every chapter that touches a literature claim cites the source in the same format.
3. **The skill pack** (`skills/autoresearch-pack/skills/citation-rigor/SKILL.md`). The skill carries the verbatim format text and the audit's enforcement rules.

The full citation list is in [`appendix_c_api_reference/glossary.md`](../appendix_c_api_reference/glossary.md). ~30 papers across deep learning, gradient boosting, calibration, cross-validation, and information retrieval.

## 10.6 The research journal

`autoresearch_results/research_journal.md` is the per-task narrative log. One entry per experiment, generated from the annotation JSON. Schema:

```markdown
## Experiment 7 — xgboost depth=10 lr=0.04

**Date:** 2026-05-15 14:24
**Champion delta:** +0.0012 (composite 0.9555 vs 0.9543)
**Verdict:** KEEP

### Diagnosis
<from annotation>

### Citations
<from annotation>

### Hypothesis
<from annotation>

### Prediction
<from annotation>

### Verdict
<from annotation>

### Learning
<from annotation>
```

The journal is a 1:1 narrative of the JSON. The JSON is the source of truth; the journal is the human-readable view. The dashboard's "Reasoning" tab renders the same content yet again, hyperlinked to the experiment row in the log.

## 10.7 The 14-section explainability audit

When a global champion lands, `framework/build_submission.py` writes a 14-section audit report to `submissions/dsbench_submission/<kind>/<slug>/audit_report.md`. The 14 sections (drawn from the autoresearch CLAUDE.md "Explainability & Auditability Report" rule):

1. Executive summary.
2. Feature importance ranking (top-20).
3. Top-N feature deep-dive (top-5 with per-fold contributions).
4. SHAP-style explanations.
5. Per-fold feature drift.
6. Calibration (reliability diagram).
7. Uncertainty quantification.
8. Prediction distribution.
9. Trade attribution (per-sample decision contributions).
10. Risk audit (failure modes, OOD detection).
11. Data pipeline audit.
12. Config dump.
13. Known limitations.
14. Deployment checklist.

Skipping any of the 14 is a regression. The `explainability-audit-14-section` skill carries the verbatim rule. Layer 3 of the four-layer audit gate ([Ch. 11](11_testing_overview.md)) checks for the 14 H2 sections in the audit report.

## 10.8 Anti-patterns we deliberately avoid

The SWE-book chapter 10 lists common documentation anti-patterns. Our counterparts:

| Anti-pattern | Mitigation |
|---|---|
| Documentation that names a TODO and never resolves it | TODOs live in ADRs' Consequences section, with a named owner; the docs/ chapters do not carry TODOs. |
| README that doesn't reflect the code | `STATUS.md` is regenerated by `_status.py`; the top-level `README.md` is short and does not duplicate `STATUS.md`. |
| Tutorials that don't run | The cohort end-to-end command (`framework/run_all.py`) is the only "tutorial"; it's tested by being the way we generate every result. |
| Inline comments that disagree with the code | Comments in `framework/runner.py` cite the audit-gate rule they enforce, not the parameters they pass. |
| Wiki pages out of date | We do not have a wiki. All documentation is in the repo and audited. |

The SWE-book's recommendation that documentation should be reviewed like code is met by `docs/` being under git, every change in PR-equivalent commits, and the link-check audit (currently manual; planned automation in [Ch. 23](../part_4_tools/23_continuous_integration.md)).

## 10.9 What this docs/ is

This `docs/` directory is the chapter-by-chapter book version of the project. It is not regenerated by scripts; it is hand-written. But it is *audited* by:

- Link-check (every internal Markdown link must resolve).
- Cross-reference check (every named principle has a single canonical home chapter).
- Citation check (every paper cited matches `appendix_c_api_reference/glossary.md`).

The book takes ~80,000 words to do its job. The unit test for "is the book correct" is: a reader who knows the SWE-book recognises every chapter parallel. The integration test is: a new operator can read [Ch. 1](../part_1_thesis/01_what_is_autoresearch_engineering.md) + [Ch. 8](08_style_guides_and_rules.md) + [Ch. 11](11_testing_overview.md) and review a commit unaided.
