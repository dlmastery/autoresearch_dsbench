# Task Description Disclosure (per-task dashboard about-block)

## When to use

- Building or upgrading a per-task dashboard.
- Hand-off to a reviewer who is opening the dashboard for the first time.
- Audit / external review preparation — the reviewer needs context on the task BEFORE the experiment table.

## The rule

Every per-task dashboard (`autoresearch_results/dashboard.html` for each task) MUST include a collapsible "About this task" block near the top of the page, BEFORE the sticky champion summary and the experiment table. Use a native HTML `<details>` element so it works without JS:

```html
<details class="about-task" open>
  <summary>About this task</summary>
  <div class="metadata-grid">
    <!-- ... -->
  </div>
</details>
```

### Required content

| Field | Source | Example |
|---|---|---|
| Task name | `task_config.json:name` | `modeloff_2014_q23_dcf` |
| Kind | `task_config.json:kind` | `analysis` |
| Problem type | `task_config.json:problem_type` | `qa_excel` |
| Task type | `task_config.json:task_type` | `multiple_choice` |
| Primary metric | `task_config.json:metric` | `exact_match_accuracy` |
| DSBench baseline | `task_config.json:dsbench_baseline` | `0.3412` |
| Current champion test score | `final_report.json:test_score` | `0.45` (if final_report has run) |
| Delta vs DSBench | `final_report.json:delta_vs_dsbench` | `+0.108` (highlight green if > 0, red if < 0) |
| Source URL | `task_config.json:url` (added by `generate_scaffolds.py`) | `https://modeloff.com/2014/q23` |
| Problem-type description | One-paragraph human-readable description of the problem family | "Excel-based DCF modelling, 4-option multiple choice, single correct answer" |
| Backbones explored | distinct values of `experiment_log.jsonl:backbone` | `xgboost, lightgbm, catboost, mlp, ft_transformer` |
| Link to CLAUDE.md | route through `dashboard/md_viewer.html?path=CLAUDE.md` | `<a target="_blank" …>` |
| Link to skill pack | absolute or relative path to `skills/autoresearch-pack/` | `<a target="_blank" …>` |

### Default state

- `<details open>` so the block is expanded on first load — the reviewer should see context immediately.
- The user can collapse it to save vertical space once they've seen the metadata.
- URL hash state (per `interactive-dashboard-design`) can persist the open/closed flag.

### Markdown link routing

Every link to a `.md` artefact (the CLAUDE.md link, any "view forensic audit" link, etc.) MUST go through `dashboard/md_viewer.html?path=…` instead of directly to the `.md` file — see the markdown-rendering rule in `interactive-dashboard-design`.

## Anti-patterns

- **Burying the about-block below the experiment table.** A first-time reviewer scrolls past it; the context they need to interpret the table is missing.
- **Hard-coding the metadata in HTML.** Source from `task_config.json` so it stays in sync as the registry updates.
- **Skipping the source URL.** The DSBench task is rooted in a public Modeloff / Kaggle problem; the reviewer must be able to click through.
- **Linking the CLAUDE.md directly (no md_viewer).** Browsers will download instead of render.
- **No link to the skill pack.** Reviewers asking "what protocol is this?" need a one-click route.
- **Showing the test score in the about-block when `final_report.py` has NOT run yet.** Render "—" or "(pending final report)" — never show a stale value.

## Implementation checklist

1. `framework/dashboard_template.html` includes the `<details class="about-task" open>` block above the sticky champion summary.
2. The dashboard JS reads `task_config.json` + `final_report.json` (when present) and fills the block on `DOMContentLoaded`.
3. The block is generated by `framework/generate_scaffolds.py` (or the dashboard's own refresh logic) so every task ships with it.
4. `dashboard/md_viewer.html` is included at the dashboard root and renders markdown via marked.js / CommonMark.
5. Validator: assert `<details class="about-task">` substring is present in every per-task `dashboard.html`.
